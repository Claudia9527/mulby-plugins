# Vibe 板块重构设计：专做 Mulby 插件的对话式编码 Agent

> 目标：把现有"四步向导（描述→契约→生成→交付）"翻转成 **一条对话主线 + 内置 Mulby 知识与能力的 agent**，
> 让**小白也能用大白话快速开发 Mulby 插件**。定位类比 Claude Code / Codex / opencode，但内置 Mulby 的脚手架、
> 契约（manifest 真相源）、构建、载入、版本、运行验证、CodeGraph 等专属能力。

## 0. 已确认的决策（来自产品负责人）

1. **意图识别**：规则优先 + 模糊时降级为"只读问答"（不擅自改代码）。
2. **左侧阶段导航**：不删除，**保留为只读进度指示**（反映当前插件生命周期，不再作为强制操作入口）。
3. **开局头脑风暴阶段**：用户提出需求时，AI 先帮用户发散、给出若干方向供选择（小白往往说不清要什么）。
4. 底层能力（脚手架/契约/构建/载入/版本/smoke/CodeGraph 注入）**全部保留复用**，只重构编排层 + UI 层。

## 1. 核心设计原则（小白友好）

- **P1 默认不动代码**：意图不明确时，先回答 / 先确认，**绝不擅自修改文件或构建**。直接根治"问个情况却被改代码并跳到交付页"。
- **P2 术语隐身**：不向小白暴露 stage / 契约 JSON / manifest 等术语；契约以可视化"**插件设定卡**"内联呈现，想看细节再展开。
- **P3 出错说人话 + 自愈**：构建/运行失败时 agent 自动尝试修复，并用一句白话解释"刚才哪里出错、怎么修的"。
- **P4 一条主线**：交互重心是**一条连续对话**，而非在多个阶段面板间跳转。

## 2. 意图模型（问题一并入问题二的关键）

每条用户消息进入编排器后，先做**意图识别**（规则优先；规则无法判定时降级为 `ask` 只读问答）：

| 意图 | 触发规则（示例关键词/形态） | 行为 | 工具集 | 是否改码/构建/进度推进 |
|---|---|---|---|---|
| `ask` 问答/查看 | "怎么/为什么/是不是/看一下/解释/现在/状态/能不能"等疑问；或无明显动作动词 | 只读回答，回复进对话区 | list_dir / read_file / grep | 否 |
| `create` 新建 | 无 createdPath 且描述了一个"要做的东西" | 头脑风暴 → 设定卡 → 脚手架+生成 | 全量 | 推进生命周期 |
| `modify` 修改/修复 | "改/修/加/去掉/换成/支持/优化/报错/没反应"等 | 改码 + 重建（现 runFollowup） | 全量 | 改码+构建 |
| `run` 运行/试用 | "运行/试一下/打开/跑一下/验证" | 打开插件 / smoke | 运行 API | 否 |
| `package` 打包 | "打包/发布/导出/inplugin" | 打包发布 | — | 否 |
| `rollback` 撤销/回滚 | "撤销/回滚/还原/退回上一版" | **二次确认后**回滚 | — | 改码+构建 |

**规则实现要点**：
- 纯前端规则函数 `classifyIntent(text, ctx)`，零延迟零成本；返回 `{ intent, confidence }`。
- `confidence` 低于阈值 → 返回 `ask`（只读兜底），并在回答里轻提示"如果你是想让我**修改**，请说'帮我改…'"。
- 每条用户消息在 UI 上打一个**可点击的意图标签**（如 `问答`/`修改`），点错了可一键改判并重跑。
- 破坏性意图（`rollback`、大范围 `modify`）执行前需在对话内出现"确认/取消"按钮。

## 3. 开局头脑风暴阶段（新增，小白核心补强）

当意图为 `create` 且需求较泛（或用户主动点"帮我想想"）时，进入头脑风暴回合：

1. AI 以一次便宜调用产出 **3-4 个候选方向**（JSON），每个含：
   - `title` 方向名（如"剪贴板图片转 Markdown"）
   - `pitch` 一句话卖点
   - `triggerPreview` 触发方式预览（关键词 / 正则 / 拖入文件…）
   - `why` 为什么适合
2. 对话内渲染为**选择卡片**，外加两个入口：
   - "**都不太对，我自己说**"（回到自由输入）
   - "**再给我几个**"（重新发散）
3. 用户选中某方向 → 该方向作为种子，AI 生成**插件设定卡**（即契约的可视化形态）。
4. 需求已经很具体的用户可跳过头脑风暴直接出设定卡，但仍可点"给我几个变体"。

> 头脑风暴让"不知道能做什么 / 说不清要什么"的小白也能两三次点击就锁定一个可落地的插件。

## 4. 新界面骨架

```
┌ 会话/项目列表 ┬──────── 对话主线（主角）────────┬ 插件状态卡（可收起）┐
│ • 剪贴板工具   │ 用户：想做个剪贴板相关的小工具    │ 📦 番茄钟  ● 已载入   │
│ • 番茄钟 ●     │ ai：给你几个方向 →               │ 触发词：番茄钟         │
│ [+ 新建会话]   │   [卡A][卡B][卡C][自己说]        │ [打开][试用][打包]     │
│                │ 用户：选 B                       │ [查看设定][版本历史]   │
│ (左侧阶段进度  │ ai：这是插件设定 → [设定卡·可编辑]│                        │
│  只读指示)     │ 用户：确认                        │ 最近改动：3 文件       │
│                │ ai：脚手架✓ 生成✓ 构建✓ 载入✓     │                        │
│                │   └ 操作卡（折叠：读/写/构建明细）│                        │
│                │ 用户：图标太丑                    │                        │
│                │ ai：已改图标 ✓ 重建 ✓             │                        │
│                │ [输入框 ｜ 当前意图：修改 ▾]      │                        │
└────────────────┴──────────────────────────────────┴────────────────────────┘
```

- **中间消息流是主角**：现在的"时间线面板"事件**内联进每条 assistant 消息**（折叠操作卡：读了哪些文件、写了哪些、构建/校验结果）。
- **左侧**：会话/项目列表（VibeSession 持久化已就绪）+ 当前插件**只读生命周期进度**（构思→已生成→已载入）。
- **右侧**：把现"交付页"零散按钮收成一张"**插件状态卡**"（路径/载入态/触发词/打开/试用/打包/查看设定/版本）。

## 5. 架构落地：复用什么 / 重构什么

### 保留复用（底层能力，基本不动）
- `runAgent` + `VIBE_TOOLS`（读/写/grep/build_check/check_conformance）
- 契约模型 `vibeContract.ts`、`manifestJson`、`normalizeContract`、`manifestToContract`
- 脚手架 `dev.createPlugin`、构建 `dev.buildPlugin`、载入 `dev.ensureLoaded`、打包 `dev.packPlugin`
- 版本 `vcs_*`、改动快照 `vibe_*`、一致性 `check_conformance`、运行验证 smoke、`injectCgContext`、`collectMulbyApiSurface`

### 重构（编排层 + UI 层）
- **新增编排器 `useVibeAgent`**：输入"一条用户消息 + 当前会话状态" → `classifyIntent` → 执行对应能力 → 产出"一条 assistant 消息（含 action 卡片）"。它取代现在散落在 `VibePanel` 里的 doPlan/doGenerate/runFollowup/runRepair… 的**串接逻辑**（这些函数本身大多可被它调用）。
- **新增只读问答 `runAsk`**：用只读工具子集 + `capabilities:['fs.read']` 跑 agent，系统提示明确"**只许查看、严禁修改/构建**"，流式回答进对话。
- **消息模型扩展**：`VibeMessage` 增加 `intent`、`actions`（操作卡明细）、`cards`（设定卡/头脑风暴选项卡）等结构，使一条消息能携带富交互。
- **UI 组件**：`MessageList` / `MessageTurn`（含 ActionCard、ContractCard、BrainstormCards、ConfirmCard）、`PluginStatusCard`、`SessionList`（左侧）。
- **阶段导航**：从可点击操作降级为只读进度条（映射 `VibeSessionState`）。

## 6. 分步实施（每步可用、可回退、保持构建绿）

| 步骤 | 内容 | 风险 | 价值 |
|---|---|---|---|
| **S1 意图层 + 只读问答** | `classifyIntent` + `runAsk`；`handleChatSend` 按意图分流（先治"问答改码"） | 低 | 立刻止血 + 打地基 |
| **S2 对话消息流** | `MessageList`/`MessageTurn` + ActionCard，timeline 内联化 | 中 | 体验转向对话 |
| **S3 头脑风暴 + 设定卡** | `runBrainstorm` + BrainstormCards + ContractCard 内联；创建流程对话化 | 中高 | 小白开局体验 |
| **S4 状态卡 + 小白引导** | PluginStatusCard、人话错误解释、危险操作确认、一键试用 | 中 | 打磨与安全 |
| **S5 清理** | 移除/降级旧阶段 UI、收敛冗余 state | 低 | 减熵 |

每一步结束都跑 `tsc --noEmit` + `npm run build` 验证。

## 7. 风险与注意

1. **向后兼容**：现有"新建插件 / AI 改造 / 工作台进入"入口必须继续工作；S1-S2 期间旧向导可与新对话并存，逐步收敛。
2. **意图误判**：规则兜底到只读，破坏性操作强制确认；意图标签可纠正。
3. **大文件**：`VibePanel.tsx`（约 2150 行）按步骤拆分，避免一次性重写。
4. **会话持久化**：复用已修好的 `useVibeSession`（含 P0 切换不污染、P1 消息持久化），新消息结构需向后兼容旧 session。
5. **token / 成本**：头脑风暴与意图分类用便宜模型/规则；只读问答限制 maxToolSteps。

## 8. 待后续确认的小决策（实施中再定）

- 意图标签纠正后是否自动重跑，还是仅改判不重跑。
- 头脑风暴默认给几个方向（建议 3）。
- 只读问答的 `maxToolSteps` 上限（建议 20）。
